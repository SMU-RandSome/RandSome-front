# 기능 개발 플로우

## 전체 흐름

```
이슈 확인 → 브랜치 생성 → 구현 (TDD) → PR 생성 → 코드 리뷰 → 머지
```

---

## 1. 이슈 확인

- GitHub Issue에서 작업할 기능을 확인한다.
- 이슈가 없으면 먼저 이슈를 생성한 후 작업을 시작한다.

---

## 2. 브랜치 생성

`main` 브랜치에서 분기한다.

```bash
git switch -c feat/#이슈번호-작업내용
# 예: feat/#12-login-page
```

**브랜치 네이밍**

| prefix | 용도 |
|--------|------|
| `feat/` | 새 기능 |
| `fix/` | 버그 수정 |
| `refactor/` | 리팩터링 (동작 변경 없음) |
| `chore/` | 빌드, 설정, 패키지 변경 |
| `style/` | 코드 포맷, 스타일 (동작 변경 없음) |

---

## 3. 구현 (TDD + Tidy First)

### TDD 사이클

```
Red → Green → Refactor
```

1. **Red**: 실패하는 테스트를 먼저 작성한다.
2. **Green**: 테스트를 통과하는 가장 단순한 코드를 작성한다.
3. **Refactor**: 중복을 제거하고 의도를 드러내도록 개선한다. 테스트는 항상 통과 상태를 유지한다.

### Tidy First 원칙

- **구조 변경(리팩터링)과 동작 변경(기능 추가)을 하나의 커밋에 섞지 않는다.**
- 코드를 정리할 때는 오직 구조만 변경하고 동작은 유지한다.
- 큰 변경을 한 번에 하지 않고, 항상 작은 단계로 나눈다.

### 커밋 컨벤션

```
<type>: <subject>

# 예시
feat: 로그인 폼 컴포넌트 추가
fix: 매칭 신청 버튼 클릭 이벤트 오류 수정
refactor: 결제 안내 컴포넌트 분리
```

| type | 용도 |
|------|------|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 리팩터링 |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드, 설정 변경 |
| `style` | 포맷, 스타일 변경 |
| `docs` | 문서 수정 |

---

## 4. PR 생성

```markdown
## 개요
<!-- 어떤 기능을 구현했는지 간략히 설명 -->

## 변경 사항
-

## 관련 이슈
close #이슈번호

## 체크리스트
- [ ] 테스트 통과
- [ ] 린트 오류 없음
```

---

## 5. 코드 리뷰 & 머지

- 리뷰어의 승인 후 `Squash and Merge`로 머지한다.
- 머지 후 작업 브랜치는 삭제한다.
